# Translate Document

기술 문서를 대상 언어로 번역하는 스킬. 마크다운 구조를 보존하면서 자연스러운 번역을 생성한다.

## 워크플로우

### 1. 입력 분석
- 번역 대상 파일 확인 (경로, 형식, 크기)
- 대상 언어 확인
- 기존 번역 파일이 있는지 확인 (업데이트 vs 신규)
- 같은 프로젝트의 기존 번역 패턴 파악 (파일명 컨벤션, 디렉토리 구조)

### 2. 컨텍스트 수집
- 같은 프로젝트의 기존 번역물이 있으면 Read로 용어 패턴 파악
- 용어집이 있으면 로드 (`references/terminology-guide.md` 참조)
- 원본 파일 전문 Read

### 3. 번역 수행

#### 마크다운 문서
- 헤딩, 리스트, 코드 블록, 링크, 이미지 참조 구조를 보존한다
- 코드 블록 내부의 코드는 번역하지 않는다. 주석(`#`, `//`, `/* */`)만 번역한다
- 파일 경로, 변수명, 함수명, CLI 명령어는 원문 유지
- 내부 앵커 링크가 헤딩 번역과 일치하도록 갱신
- YAML frontmatter의 값은 번역하되 키는 유지

#### JSON i18n 메시지
- 키는 원문 유지, 값만 번역
- 플레이스홀더(`{name}`, `{{count}}` 등)는 원문 그대로 유지
- 복수형 규칙이 언어마다 다르므로 대상 언어의 복수형 체계에 맞게 조정

#### 파일 저장 규칙
- 기존 번역 파일의 네이밍 패턴을 따른다
  - 예: `README.md` → `README-ko.md`, `docs/en/` → `docs/ko/`
- 패턴이 없으면 `{filename}.{lang}.{ext}` 또는 `{lang}/{filename}` 사용
- 출력 파일 상단에 원본 파일 경로를 주석으로 남기지 않는다 (기존 번역물에 없는 경우)

### 4. 자체 검증
- 번역 후 원문과 줄 수를 비교하여 큰 차이가 없는지 확인 (30% 이상 차이면 재검토)
- 마크다운 구문이 깨지지 않았는지 확인 (헤딩 수, 코드 블록 수 매칭)
- 코드 블록 내 코드가 변경되지 않았는지 확인

## 언어별 가이드

필요시 `references/terminology-guide.md`를 Read하여 프로젝트 특화 용어집을 참조한다.

### 공통 기술 용어 처리
- 업계 표준 번역이 있는 용어는 해당 번역 사용
- 번역어가 아직 정착되지 않은 용어는 원어 병기: `하네스(Harness)`
- 약어(API, CLI, SDK 등)는 번역하지 않는다
- 브랜드명(Claude, GitHub 등)은 번역하지 않는다
